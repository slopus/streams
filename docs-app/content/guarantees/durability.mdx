# Durability Classes

Durability in streams is a **per-box** decision, not a server-wide mode. Each box is one of
three **commit classes** — `memory`, `disk`, or `fsync` — chosen at create time, that trade
write latency against crash survival. A RAM-only cache box, a group-committed pub/sub feed,
and an `fsync`-gated job queue coexist in one process, and each pays for exactly the guarantee
it uses.

The class is resolved once when the box is created (`durability_class()`), reported on every
box-state and box-create response, and never reinterpreted at runtime.

## The three classes

<Callout type="info">
  The defining promise across all three: **an acked write is published, and nothing visible
  is ever un-durable.** A write that fails to commit publishes nothing — there is no
  readable-but-not-yet-durable state. What differs between classes is *when* the ack fires and
  *what a crash costs*, not whether an acked write is real.
</Callout>

### `memory`

_"disk-like but best-effort."_ A `memory` box takes the **same** group-committed WAL write
**and** recovery path as `disk` and is **fully queryable** (getState / getDifference / SSE) —
but it carries **NO durability guarantee**. The ack is never `fsync`-gated, so `fsync_ms` is
`0` (the fastest path).

After a restart its records **MAY survive OR be lost** — recovery is **gradual / best-effort**:
it does **not** block readiness and does **not** guarantee completeness or emptiness. The box
**config always persists** (it is a control frame in the WAL). The one hard bound is
no-fabrication / no-future-seq: a recovered record is always one that was actually written, and
the head never hands out a seq past what was acked (a best-effort restart may legitimately
regress the head if the un-fsynced tail was lost — there is no durable seq reservation).

Effectively `disk` minus the durability promise. Reach for `memory` for caches, scratch state,
and ephemeral fan-out where occasional loss is an acceptable trade for the lowest latency. It is
reachable **only** by setting `durability:"memory"` explicitly — no legacy bool produces it.

### `disk`

Records are written to the WAL and **group-committed** — no per-write `fsync` — so `fsync_ms`
is `0` (the fast path). The write is acked **as soon as its frame is enqueued** to the single
WAL writer; the ack is **not** fsync-gated. The writer then group-commits and `fdatasync`s the
batch shortly after.

A crash loses the **un-fsynced tail** — the frames that were enqueued but not yet
group-fsynced. Everything older is recovered by WAL replay. This is the default and the
pub/sub workhorse: durable enough that a clean shutdown loses nothing, fast enough that the
common feed isn't paying for a guarantee it doesn't want. `disk` is today's `durable:false`.

### `fsync`

The ack is **`fsync`-gated**: it is held until the WAL frame is durably synced, so the response
carries a real `fsync_ms`. The write **survives any crash** — an acked write is always
recovered by WAL replay.

This is the class for job queues, financial events, and anything where an acknowledgment is a
promise you can never take back. It is today's `durable:true`. The per-write cost is the disk's
fsync latency, but **adaptive group commit** amortizes one `fsync` across a whole batch of
concurrent durable writers, so the per-event cost approaches a sequential disk append rather
than one fsync per record. See [WAL & group commit](/how-it-works/wal-and-group-commit).

<Callout type="warning">
  `fsync` durability is bounded by the disk's `fdatasync` latency. On a laptop's APFS NVMe
  that floor is roughly **5 ms**; server-grade NVMe is typically **50–500 µs** (about 10×
  faster). This is a hardware property, not a design cost — group commit hides it under
  concurrency, but a single serial durable writer pays the floor per write. See
  [Performance](/how-it-works/performance).
</Callout>

## Comparison

| Class | WAL? | Ack fires | Survives a crash? | `fsync_ms` |
|---|---|---|---|---|
| `memory` | yes, group-committed (same path as `disk`) | immediately, **not** fsync-gated | **best-effort** — records MAY survive OR be lost (no guarantee); config always persists | `0` |
| `disk` | yes, group-committed (no per-write fsync) | on WAL-frame enqueue, **not** fsync-gated | yes, **minus the un-fsynced tail** | `0` |
| `fsync` | yes, fsync-gated | after the group `fsync` | **yes, any crash** — an acked write is always recovered | real value |

## Choosing and configuring a class

Set the class with the `durability` field at create time:

```bash
# An fsync-gated, unbounded job queue: every ack is a durable promise.
curl -X PUT localhost:4000/v0/boxes/jobs \
  -H 'content-type: application/json' \
  -d '{ "durability": "fsync", "cap_records": 0, "ttl_ms": 0 }'
```

```json
{ "box": "jobs", "created": true,
  "config": { "ttl_ms": 0, "cap_records": 0, "cap_bytes": 0, "discard": "old",
              "durable": true, "durability": "fsync", "auto_create": true,
              "idempotency_window_ms": 120000, "dedupe_node": true },
  "performance": { "server_total_ms": 0.22 } }
```

The class defaults to `disk` when neither `durability` nor `durable` is set, so an
auto-created box (one materialized lazily on first write) is `disk`. To get `memory`, you
must say so explicitly.

### The legacy `durable` bool

`durable` is a back-compat alias kept so older clients keep working:

- `durable: true` ⇒ `fsync`
- `durable: false` ⇒ `disk`
- `memory` is reachable **only** via `durability:"memory"`

An explicit `durability` always wins over `durable`. On the way out, `durable` is normalized
to `durable == (durability == "fsync")`, so a legacy client reading the `durable` boolean on a
box-state response still sees the right value regardless of how the box was configured.
Internally, `is_durable()` is simply `class == "fsync"`.

<Callout type="info">
  Durability only governs **persistence across a crash or restart**. It is independent of
  retention: `ttl_ms` and the `cap_records`/`cap_bytes` caps still apply to every class, and
  involuntary eviction or expiry of live records always surfaces as a
  [tombstone](/guarantees/tombstones) — even on a `memory` box, for as long as that box lives.
</Callout>

## Routers and dead-letter honor the destination class

A router-forwarded copy and a dead-lettered job are ordinary appends into their destination
box, so they take on the **destination box's** commit class — not the source's:

- A `fsync` destination advances the router's forward cursor only **after** the destination
  fsyncs the forwarded record.
- A `disk` destination writes the forwarded copy to the WAL, group-committed.
- A `memory` destination takes the **same best-effort path** as any `memory` write — the
  forwarded copy is group-committed to the WAL like `disk` but carries **no durability
  guarantee**, exactly like a direct write to that box.

So if you route an `fsync` source into a `memory` destination, the forwarded copies are
best-effort — they **MAY survive OR be lost** on a restart (the dest config always persists),
while the durable source recovers in full. The durability of the copy follows where it lands,
not where it came from. Size and class your destinations deliberately. See
[Routers](/api/routers) and the [Multi-master guide](/guides/multi-master).

<Callout type="info">
  Router forwarding is at-least-once via the source log, but a persistently-failing or slow
  durable destination is currently held as **backpressure with no background retry driver
  yet** — a stuck destination can lag. Queue lease durability is best-effort
  (`leases_durable` defaults `false`); a transient WAL error on a lease append degrades to the
  queue's baseline at-least-once rather than losing or duplicating work. See the maturity notes
  in the [Introduction](/getting-started).
</Callout>

## How a crash actually recovers

On restart the engine loads the latest snapshot, replays the WAL forward from the snapshot's
checkpoint, truncates a torn tail (any frame whose length runs past EOF or whose XXH3-64
checksum fails to verify), and reclaims orphaned segments. An acked durable write is, by
construction, a **complete, checksum-valid WAL frame** — so it is never lost. The only data a
crash can cost is what a class explicitly does not promise: a `memory` box's records, or a
`disk` box's un-fsynced tail. Both surface to consumers as ordinary eviction-style gaps. See
[Recovery](/how-it-works/recovery).
