```bash
# Write a record. The server numbers it and creates the topic on first write.
curl -X POST localhost:4000/v0/topics/jobs \
  -H 'content-type: application/json' \
  -d '{ "records": [ { "data": { "url": "s3://b/a.png" } } ] }'
# → { "topic": "jobs", "seqs": [1], "head_seq": 1, ... }

# Read everything after sequence 0. The number IS the cursor.
curl -X POST localhost:4000/v0/topics/jobs/diff -d '{ "from_seq": 0 }'
# → { "records": [ { "$seq": 1, "$ts": 1748470000123, "data": { "url": "s3://b/a.png" } } ],
#     "next_from_seq": 1, "head_seq": 1, "caught_up": true }
```

<section className="section">

<span className="section-eyebrow">The API</span>
<h2 className="section-title">The whole thing is a handful of HTTP calls</h2>
<p className="section-lede">
streams is a log you <code>POST</code> events to and read back by number. The server
assigns each event a monotonic <code>seq</code>; reading is just "give me everything after
this number." That sentence is the entire model — here is every verb it expands into.
</p>

<div className="concept-grid">
  <div className="concept"><div className="c-name">create</div><p className="c-desc">A topic appears on first write — or <code>PUT /v0/topics/:topic</code> to set config up front. Nothing to provision.</p></div>
  <div className="concept"><div className="c-name">write</div><p className="c-desc"><code>POST /v0/topics/:topic</code> with a batch of records. The server numbers each one and returns the assigned seqs.</p></div>
  <div className="concept"><div className="c-name">read</div><p className="c-desc"><code>POST …/diff</code> with <code>from_seq</code>. Get the records after it plus the next cursor. The seq is the cursor — advancing it is the ack.</p></div>
  <div className="concept"><div className="c-name">watch</div><p className="c-desc"><code>POST /v0/watch</code> then open the SSE stream. Follow many topics live over one resumable connection, woken on append.</p></div>
  <div className="concept"><div className="c-name">delete</div><p className="c-desc"><code>POST …/delete</code> by seq range and/or tag. Permanent, immediate, point-in-time — it never touches future records.</p></div>
  <div className="concept"><div className="c-name">fan out</div><p className="c-desc"><code>PUT /v0/routers/:r</code> to copy every record from one topic into another, server-side, preserving origin.</p></div>
</div>

<p className="section-lede" style={{ marginTop: '1.4rem' }}>
No connection modes, no binary framing, no consumer-group bookkeeping on the server.
Read it at three levels of depth on the <a href={withBase("/api")}>API reference</a>.
</p>

<section className="section">

<span className="section-eyebrow">Durability you dial in</span>
<h2 className="section-title">Choose what an "ok" means — per topic</h2>
<p className="section-lede">
Every write travels the same road: your process → the kernel → the physical disk. The only
question is how far along streams waits before it tells you "ok." You answer that
<strong>per topic</strong>, so a throwaway cache and a financial ledger live happily in one
process.
</p>

<div className="dura-grid">
  <div className="dura">
    <span className="d-tag">memory</span>
    <h4>Fastest — no promise</h4>
    <p>Acked the instant streams accepts it, with <strong>no durability guarantee</strong>: after a restart its records may survive or may be gone. For caches, scratch, and ephemeral fan-out where losing a little is fine.</p>
  </div>
  <div className="dura">
    <span className="d-tag">disk</span>
    <h4>Survives a crash</h4>
    <p>Written to a group-committed write-ahead log and handed off toward the kernel before the ack. <strong>Survives a process crash</strong>, minus the most recent not-yet-flushed tail. The pub/sub default.</p>
  </div>
  <div className="dura">
    <span className="d-tag">fsync</span>
    <h4>Survives power loss</h4>
    <p>The ack waits for the data to be <strong>fsynced to the platter</strong>. An acknowledged write survives <em>any</em> crash, including power loss, and is replayed from the WAL on restart. For queues and ledgers.</p>
  </div>
</div>

<p className="section-lede" style={{ marginTop: '1.4rem' }}>
And whatever the class, an acked write is published and nothing visible is ever un-durable.

<section className="section">

<span className="section-eyebrow">Mental model</span>
<h2 className="section-title">Five nouns and you know the system</h2>
<p className="section-lede">
streams is deliberately small. Learn five concepts and every endpoint is just one of them
doing its job.
</p>

<div className="concept-grid">
  <div className="concept"><div className="c-name">Topic</div><p className="c-desc">A named, append-only log of records ordered by a monotonic <code>seq</code>. Created lazily on first write.</p></div>
  <div className="concept"><div className="c-name">Record</div><p className="c-desc">One immutable event. The server assigns <code>$seq</code> and <code>$ts</code>; you supply opaque <code>data</code>, an optional <code>tag</code> and <code>node</code>.</p></div>
  <div className="concept"><div className="c-name">seq</div><p className="c-desc">The cursor. Reads are "give me everything after <code>from_seq</code>." The client owns its position; advancing it is the ack.</p></div>
  <div className="concept"><div className="c-name">Router</div><p className="c-desc">A server-side <code>source → dest</code> forwarding rule. Fan-out that preserves origin node, so N nodes mirror without echo.</p></div>
  <div className="concept"><div className="c-name">Tombstone</div><p className="c-desc">The explicit "you missed data" signal. Cap/TTL loss returns an in-band gap range — at HTTP 200, never silently.</p></div>
</div>

<p className="section-lede" style={{ marginTop: '1.4rem' }}>
The load-bearing rule connecting them: <strong>loss you didn't ask for is always announced;
removal you did ask for is silent.</strong> Read it in full in the
<a href={withBase("/getting-started/mental-model")}>mental model</a>.
</p>

</section>
