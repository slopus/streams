# Quickstart

This walks through the whole product in a few `curl` commands: run the server, write
records, read them back from a cursor, watch a live SSE stream, fan out with a router,
and delete. Everything is plain HTTP and JSON — no client library required.

<Callout type="info">
  The examples assume the binary is running on `localhost:4000` with auth disabled (dev
  mode). That is the default on a loopback bind. See [Install & Run](/getting-started/installation)
  and [Configuration](/deployment/configuration) for production settings.
</Callout>

<Steps>

### Run the server

<Tabs items={['Binary', 'Docker']}>
  <Tabs.Tab>
```bash
# Build the single binary (Rust toolchain required) and run it.
cargo build --release
./target/release/streams
# Listening on 127.0.0.1:4000 — auth disabled (dev mode).
# WAL + segments + snapshots live under ./streams-data
```
  </Tabs.Tab>
  <Tabs.Tab>
```bash
# The image binds 0.0.0.0:4000 and refuses to start on a non-loopback bind with no
# keys — so pass STREAMS_ALLOW_INSECURE_NO_AUTH=1 for a local-only run.
docker run --rm -p 4000:4000 -v streams-data:/data \
  -e STREAMS_ALLOW_INSECURE_NO_AUTH=1 \
  ghcr.io/slopus/streams:latest
```
  </Tabs.Tab>
</Tabs>

### Create a topic (optional)

A topic is created lazily on first write, so this step is optional — but it lets you set
config up front. Here we make a durable, unbounded queue.

```bash
curl -X PUT localhost:4000/v0/topics/jobs \
  -H 'content-type: application/json' \
  -d '{ "durable": true, "cap_records": 1000000, "ttl_ms": 0 }'
```

```json
{ "topic": "jobs", "created": true,
  "config": { "ttl_ms": 0, "cap_records": 1000000, "cap_bytes": 0, "discard": "old",
              "durable": true, "durability": "fsync", "auto_create": true,
              "idempotency_window_ms": 120000, "dedupe_node": true },
  "performance": { "server_total_ms": 0.22 } }
```

### Write records

The server assigns each record a `$seq` (a monotonic `u64`) and a `$ts`. A single `POST`
can carry up to 10,000 records, committed atomically.

```bash
curl -X POST localhost:4000/v0/topics/jobs \
  -H 'content-type: application/json' \
  -d '{ "node": "worker-eu-1",
        "records": [
          { "data": { "url": "s3://b/a.png", "w": 256 }, "tag": "tenant42:job-9001" },
          { "data": { "url": "s3://b/b.png", "w": 512 }, "tag": "tenant42:job-9002" }
        ] }'
```

```json
{ "topic": "jobs", "first_seq": 1, "last_seq": 2, "seqs": [1, 2],
  "head_seq": 2, "count": 2, "created": false, "deduped": false,
  "performance": { "server_total_ms": 0.62, "fsync_ms": 0.39 } }
```

### Read from a cursor

The core consume operation. Pass a `from_seq` cursor; get the records after it, a
continuation cursor, and any tombstone. The monotonic `seq` **is** the cursor — there is
no opaque token. Advancing your stored `from_seq` is the ack.

```bash
curl -X POST localhost:4000/v0/topics/jobs/diff \
  -H 'content-type: application/json' \
  -d '{ "from_seq": 0, "limit": 500, "node": "worker-eu-1" }'
```

```json
{ "topic": "jobs",
  "records": [
    { "$seq": 1, "$ts": 1748470000123, "data": { "url": "s3://b/a.png", "w": 256 } },
    { "$seq": 2, "$ts": 1748470000140, "data": { "url": "s3://b/b.png", "w": 512 } }
  ],
  "next_from_seq": 2, "head_seq": 2, "earliest_seq": 1,
  "caught_up": true, "tombstone": null, "lag": 0,
  "performance": { "server_total_ms": 0.30 } }
```

<Callout type="info">
  Records written by `worker-eu-1` are filtered out when `worker-eu-1` reads them back —
  [node loop-prevention](/guarantees/node-loop-prevention). The cursor still advances past
  them, so `caught_up` — not `records.length` — is the reliable "no more right now" signal.
</Callout>

### Watch many topics over one SSE stream

`POST /v0/watch` creates a session and returns a `wid`; a `GET` on that `wid` opens the
`EventSource`-compatible stream.

```bash
# 1. Create the watch session (carries the full subscription).
curl -X POST localhost:4000/v0/watch \
  -H 'content-type: application/json' \
  -d '{ "node": "worker-eu-1",
        "topics": { "jobs": { "from_seq": 0 }, "events": { "tail": true } } }'
# → { "wid": "wid_BuRguGorNdVFWNQULz-rrw", "stream_url": "/v0/watch/wid_BuRguGorNdVFWNQULz-rrw", ... }

# 2. Open the stream.
curl -N localhost:4000/v0/watch/wid_BuRguGorNdVFWNQULz-rrw
```

```text
retry: 2000

id: eyJqb2JzIjoxfQ
event: record
data: {"topic":"jobs","records":[{"$seq":1,"$ts":1748470000123,"data":{"url":"s3://b/a.png"}}],"from_seq":0,"to_seq":1,"head_seq":2}

id: eyJqb2JzIjoyfQ
event: caught-up
data: {"topic":"jobs","head_seq":2}

: hb
```

### Fan out with a router

A router copies every record appended to `source` into `dest` — at-least-once,
per-source FIFO. The origin `$node` rides through untouched, so symmetric nodes mirror
without echo.

```bash
curl -X PUT localhost:4000/v0/routers/jobs-to-audit \
  -H 'content-type: application/json' \
  -d '{ "source": "jobs", "dest": "audit" }'
```

### Delete records (permanent, point-in-time)

A delete removes records that exist **right now**, by `seq` range and/or `tag` match. It
is permanent, silent (never a tombstone), effective immediately on all reads, and
point-in-time (future records with the same tag are never affected).

```bash
# Cancel one job by exact tag match.
curl -X POST localhost:4000/v0/topics/jobs/delete \
  -H 'content-type: application/json' \
  -d '{ "match": ["tag", "Eq", "tenant42:job-9001"] }'
```

```json
{ "topic": "jobs", "deleted": 1, "earliest_seq": 2, "head_seq": 2, "count": 1,
  "performance": { "server_total_ms": 0.12 } }
```
