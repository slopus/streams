# Pub/Sub

Pub/sub on streams is a **small capped, TTL'd feed topic** with [routers](/api/routers) fanning every record out to one topic per subscriber, and subscribers **tailing over SSE** with `tail: true`. Memory stays bounded because the feed and the subscriber topics have a small `cap_records` and a short `ttl_ms`; a subscriber that falls behind the retention window doesn't break вҖ” it gets an explicit [tombstone](/guarantees/tombstones) frame naming the exact gap, then resumes. This is the weak-guarantee, low-latency pattern: fast fan-out, bounded resources, gaps tolerated and *signalled*, never silent.

It is deliberately the opposite of the [durable-streams](/guides/durable-streams) recipe. There you make eviction impossible; here you embrace it for bounded memory, and rely on the gap signal to keep slow subscribers honest.

<Callout type="info">
  Use the `disk` durability class (the default, `durable: false`) for pub/sub. A
  group-committed WAL keeps latency near the ~1 ms target without paying a per-write
  `fsync`, and a crash only loses the un-fsynced tail вҖ” acceptable for a feed whose whole
  point is recency. Use `durability: "memory"` for the best-effort class вҖ” the same
  group-committed WAL path as `disk` but with no durability guarantee (after a restart its
  records may survive or be lost). See [Durability classes](/guarantees/durability).
</Callout>

```text
                 в”Ӯ ttl  в”Ӯ в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв–әв”Ӯ  sub-b  в”Ӯ в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв–ә (watch sub-b)

```

A publisher writes once to `feed`. Routers copy each record into `sub-a`, `sub-b`, вҖҰ вҖ” one topic per subscriber. Each subscriber watches *its own* topic over SSE. Giving each subscriber its own topic (rather than sharing one) means each keeps an independent cursor, an independent retention window, and an independent slow-consumer fate.

## Build it

<Steps>

### Create the feed with a small cap + TTL

A bounded feed: keep at most `cap_records` records, expire anything older than `ttl_ms`. `discard: "old"` (the default) evicts the oldest when full вҖ” that is the pub/sub-friendly policy.

```bash
curl -X PUT localhost:4000/v0/topics/feed \
  -H 'content-type: application/json' \
  -d '{ "ttl_ms": 5000, "cap_records": 10000, "discard": "old", "durable": false }'
```

```json
# вҶ’
{ "topic": "feed", "created": true,
  "config": { "ttl_ms": 5000, "cap_records": 10000, "cap_bytes": 0, "discard": "old",
              "durable": false, "durability": "disk", "auto_create": true,
              "idempotency_window_ms": 120000, "dedupe_node": true },
  "performance": { "server_total_ms": 0.21 } }
```

### Fan out with one router per subscriber

A router copies every record appended to `feed` into a per-subscriber topic, at-least-once and per-source FIFO. The subscriber topics auto-create (lazily) on first forward, inheriting defaults вҖ” give them their own small cap+TTL up front if you want a different retention than the feed.

```bash
curl -X PUT localhost:4000/v0/topics/sub-a -d '{ "ttl_ms": 5000, "cap_records": 10000 }'
curl -X PUT localhost:4000/v0/topics/sub-b -d '{ "ttl_ms": 5000, "cap_records": 10000 }'

curl -X PUT localhost:4000/v0/routers/feed-to-a -d '{ "source": "feed", "dest": "sub-a" }'
curl -X PUT localhost:4000/v0/routers/feed-to-b -d '{ "source": "feed", "dest": "sub-b" }'
```

```json
# вҶ’ (for each router)
{ "router": "feed-to-a", "created": true, "source": "feed", "dest": "sub-a",
  "preserve_node": true, "preserve_tag": true, "filter": null, "allow_cycle": false,
  "performance": { "server_total_ms": 0.20 } }
```

<Callout type="info">
  A forward obeys the **destination** topic's config: with `discard: "old"`, a full `sub-a`
  evicts its oldest and the forward succeeds (a lagging subscriber A loses the oldest, gets a
  tombstone, and B is unaffected). Routers can also carry a `filter` (the same tuple language
  as deletes) to forward only matching records вҖ” e.g. `["tag","Glob","public:*"]` for a
  per-subscriber topic filter. See [Routers](/api/routers).
</Callout>

### Publish

The publisher writes to `feed`; the routers do the fan-out. Tag records if subscribers will filter or you'll want to cancel by tag.

```bash
curl -X POST localhost:4000/v0/topics/feed \
  -H 'content-type: application/json' \
  -d '{ "records": [
          { "data": { "type": "price", "sym": "ACME", "px": 42.10 }, "tag": "public:prices" }
        ] }'
```

```json
# вҶ’
{ "topic": "feed", "first_seq": 1, "last_seq": 1, "seqs": [1],
  "head_seq": 1, "count": 1, "created": false, "deduped": false,
  "performance": { "server_total_ms": 0.18 } }
```

### Subscribe by tailing over SSE

Each subscriber watches its own topic with `tail: true`, which starts at the topic's current head вҖ” only records published *after* subscribe (the SSE analog of Redis `XREAD $`). Create the session, then open the stream.

```bash
# 1. Create the watch session.
curl -X POST localhost:4000/v0/watch \
  -H 'content-type: application/json' \
  -d '{ "topics": { "sub-a": { "tail": true } } }'
# вҶ’ { "wid": "wid_BuRguGorNdVFWNQULz-rrw", "stream_url": "/v0/watch/wid_BuRguGorNdVFWNQULz-rrw", ... }

# 2. Open the stream.
curl -N localhost:4000/v0/watch/wid_BuRguGorNdVFWNQULz-rrw
```

```text
retry: 2000

id: eyJzdWItYSI6MX0
event: record
data: {"topic":"sub-a","records":[{"$seq":1,"$ts":1748470000123,"data":{"type":"price","sym":"ACME","px":42.10}}],"from_seq":0,"to_seq":1,"head_seq":1}

: hb
```

</Steps>

A subscriber can watch **many** topics over the one SSE connection (up to 256), each with its own cursor вҖ” so a single client tailing several topics needs only one stream. The `id:` on each data frame encodes the whole per-topic cursor map; on reconnect the client passes it as `Last-Event-ID` to resume exactly.

## Tolerating gaps вҖ” the tombstone

This is the load-bearing part of weak-guarantee pub/sub. Because the subscriber topics are small and TTL'd, a slow subscriber *will* eventually fall behind the retention window. When that happens streams does **not** silently drop the missed records вҖ” it emits a tombstone frame naming the exact range, then resumes from the new floor:

```text
id: eyJzdWItYSI6ODMwMDB9
event: tombstone
data: {"topic":"sub-a","reason":"cap","gap_from":80000,"gap_to":83000,"earliest_seq":83001,"head_seq":88130}
```

The subscriber handles `record` and `tombstone` uniformly: process records, and on a tombstone log/account for the missed `[gap_from, gap_to]` range (and re-fetch state from your source of truth if you need to). `reason` is `cap` (evicted for capacity), `ttl` (expired), `mixed` (both), or `from_seq_too_old` (the requested cursor was already below the floor at connect time). The frame's `id:` already advances the cursor past the gap, so a resume after it is correct.

<Callout type="warning">
  The tombstone is the contract that makes "tolerate gaps" safe: involuntary cap/TTL loss is
  **always** signalled in-band at HTTP 200, never a silent skip. A subscriber that ignores
  tombstones is choosing to drop the missed range; a subscriber that handles them can fall
  arbitrarily far behind and still know precisely what it missed. See
  [Tombstones](/guarantees/tombstones).
</Callout>

## Liveness and slow consumers

- **Heartbeats** keep the connection alive when idle. After `heartbeat_ms` (default 15000, clamped `[1000, 60000]`) with no data, the server sends a bare `: hb` comment вҖ” it carries no payload and no `id:`, so it never perturbs your resume cursor. Suppressed whenever real data went out in the window. Arm a client watchdog at roughly `2 Г— heartbeat_ms` and reconnect on timeout.
- **Slow consumers** have a bounded outbound queue; while it can't drain, records coalesce into larger, fewer batches. If a subscriber stays stuck, a lossy (capped/TTL'd) topic degrades to a tombstone on resume вҖ” exactly the gap signal above вҖ” while the rest of the stream keeps flowing. The session survives the disconnect for `session_ttl_ms` (default 300000) so the client resumes with no loss beyond the topic's own cap/TTL.

## Notes & variations

- **Lightweight tailing.** Set `include_data: false` on the watch to receive only `$seq`/`$ts`/`$tag`/`$node` metadata per record вҖ” handy for a "something changed, go fetch it" notification feed.
- **One shared topic instead of per-subscriber topics.** You *can* point every subscriber at `feed` directly and skip the routers вҖ” but then all subscribers share one retention window, and a single slow subscriber's lag is measured against the same cap as everyone else. Per-subscriber topics isolate each subscriber's fate; that isolation is the reason to use routers here.
- **Bounded memory across the whole topology.** Each topic trims itself via cap+TTL. To put a hard ceiling on total retained bytes server-wide, set `STREAMS_MAX_TOTAL_BYTES` вҖ” a write past it returns `429 throttled`. See [Configuration](/deployment/configuration).
